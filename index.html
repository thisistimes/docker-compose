ในโปรเจกต์ตัวอย่างมีไฟล์อะไรบ้าง?

มือใหม่มักงงว่าไฟล์ไหนทำหน้าที่อะไร ตารางนี้คือภาพรวมแบบง่ายที่สุด

ไฟล์หน้าที่เปรียบเทียบง่าย ๆ
package.jsonบอกชื่อโปรเจกต์, คำสั่ง npm, และ library ที่ใช้ใบรายการของที่ต้องซื้อ
package-lock.jsonล็อกเวอร์ชัน library ให้ติดตั้งเหมือนเดิมทุกเครื่องใบเสร็จที่ระบุรุ่นเป๊ะ ๆ
app.jsโค้ด API ตัวอย่างตัวแอปจริง
Dockerfileสูตรสร้าง image ของ APIสูตรทำกล่องแอป
docker-compose.ymlบอกให้เปิด frontend, api, mongodb พร้อมกันรีโมตเปิดทั้งระบบ

package.json ตัวอย่าง

{
  "name": "docker-demo-api",
  "version": "1.0.0",
  "scripts": {
    "start": "node app.js",
    "dev": "nodemon app.js"
  },
  "dependencies": {
    "express": "^4.18.2",
    "mongodb": "^6.3.0"
  },
  "devDependencies": {
    "nodemon": "^3.0.2"
  }
}

อ่านยังไง?

scripts คือคำสั่งลัด เช่น npm start จะไปเรียก node app.js
dependencies คือของที่แอปต้องใช้ตอนรันจริง เช่น express, mongodb
devDependencies คือของที่ใช้ตอนพัฒนา เช่น nodemon
package*.json ใน Dockerfile หมายถึง copy ทั้ง package.json และ package-lock.json

Dockerfile = สูตรสร้างกล่อง API

Docker จะอ่านไฟล์นี้จากบนลงล่าง แล้วสร้าง image ออกมา เหมือนทำอาหารตามสูตรทีละขั้น

# 1) เริ่มจากกล่องที่มี Node.js ให้แล้ว
FROM node:20-alpine

# 2) เข้าไปทำงานในโฟลเดอร์ /app
WORKDIR /app

# 3) เอาไฟล์รายชื่อ package เข้าไปก่อน
COPY package*.json ./

# 4) ติดตั้ง package ตาม package-lock.json
RUN npm ci

# 5) เอาโค้ดทั้งหมดเข้าไปในกล่อง
COPY . .

# 6) บอกว่า API ใช้ port 8080 ใน container
EXPOSE 8080

# 7) ตอนเปิด container ให้รันคำสั่งนี้
CMD ["npm", "start"]
FROM = เริ่มจาก image สำเร็จรูป เช่น Node.js
WORKDIR = เลือกโฟลเดอร์ทำงานใน container
COPY package*.json ./ = เอา package file เข้าไปก่อน เพื่อให้ Docker cache ขั้นติดตั้ง package ได้ดี
RUN npm ci = ติดตั้ง dependencies ตอน build image
COPY . . = copy code ทั้งโปรเจกต์เข้าไป
EXPOSE = บอก port ภายใน container ยังไม่ใช่ port ที่เปิดบนเครื่องเรา
CMD = คำสั่งเริ่มต้นตอน container ทำงาน
จำง่าย ๆ: RUN ทำตอนสร้าง image, ส่วน CMD ทำตอนเปิด container

docker-compose.yml = รีโมตเปิดทั้งระบบ

ไฟล์นี้บอก Docker ว่าระบบเรามีกี่กล่อง แต่ละกล่องชื่ออะไร เปิด port ไหน และคุยกันยังไง

services:
  frontend:
    image: nginx:alpine
    ports:
      - "3000:80"
    depends_on:
      - api

  api:
    build: ./api
    ports:
      - "8080:8080"
    environment:
      MONGO_URL: mongodb://mongodb:27017/demo
    depends_on:
      - mongodb

  mongodb:
    image: mongo:7
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db

volumes:
  mongo_data:
services = รายชื่อกล่องทั้งหมดในระบบ
frontend = กล่องหน้าเว็บ เปิดให้เราเข้าผ่าน localhost:3000
api = กล่อง backend build จากโฟลเดอร์ ./api
database service = กล่องฐานข้อมูล อาจเป็น MongoDB, PostgreSQL, MySQL หรือ Redis ก็ได้
ports "3000:80" = ซ้ายคือ port เครื่องเรา ขวาคือ port ใน container
environment = ส่งค่าตั้งค่าเข้า container เช่น URL, username, password หรือชื่อ database
volumes = เก็บข้อมูล DB ให้ไม่หายเมื่อปิด container
จุดสำคัญ: ใน Compose container คุยกันด้วย ชื่อ service เช่น mongodb, postgres, mysql หรือ redis ไม่ใช่ localhost

เลือกฐานข้อมูลที่อยากจำลอง

Docker Compose ไม่ได้ใช้ได้แค่ MongoDB เท่านั้น เราสามารถเปลี่ยน “กล่องฐานข้อมูล” เป็นตัวอื่นได้ โดยหลัก ๆ เปลี่ยนแค่ service ของ database และ connection string ที่ API ใช้ต่อเข้าไป


        

      

      
จำง่าย ๆ: ถ้าเปลี่ยน DB ให้ดู 4 จุดนี้เสมอ — image, ports, environment, และ volumes. ส่วน API ต้องเปลี่ยน connection string ให้ตรงกับ service name ของ DB นั้น ๆ

ส่วนนี้ช่วยอ่าน error แบบไม่ตกใจ แล้วรู้ว่าต้องแก้ตรงไหน

      

        
❌ port is already allocated
แปลว่า port นี้มีโปรแกรมอื่นใช้อยู่ เช่น 3000 ซ้ำ
แก้: เปลี่ยนซ้ายมือใน ports เช่น "3001:80"

        
❌ API ต่อ DB ไม่ได้
ใน container ห้ามใช้ localhost เพื่อเรียก DB อีก container
แก้: ใช้ชื่อ service เช่น mongodb, postgres, mysql หรือ redis ตามชื่อใน compose

        
❌ npm install failed
อาจเกิดจาก package-lock ไม่ตรง หรือ package มีปัญหา
แก้: เช็ก package.json, package-lock.json แล้วลอง build ใหม่

        
❌ แก้โค้ดแล้วไม่เปลี่ยน
image เก่าอาจยังถูกใช้ หรือไม่ได้ mount volume สำหรับ dev
แก้: รัน docker compose up -d --build ใหม่

      

    



    

      

        
Cheat Sheet: กดเปิดอ่านทีละคำสั่ง

        
หน้านี้พับเก็บไว้ก่อนเพื่อไม่ให้รก อยากรู้คำสั่งไหนก็กดเปิดอ่านเฉพาะอันนั้นได้เลย

      


      
🚀 เริ่ม/หยุดระบบ

      

        
docker compose up
อ่านไฟล์ docker-compose.yml แล้วเปิดทุก service เช่น frontend, api, database พร้อมแสดง log บน terminal
ใช้เมื่อ: อยากเห็น log สด ๆ ระหว่างระบบกำลังเปิด

        
docker compose up -d
-d คือ detached mode เปิด container ทิ้งไว้ด้านหลัง ไม่ล็อกหน้าจอ terminal
ใช้เมื่อ: อยากเปิดระบบทิ้งไว้ แล้วพิมพ์คำสั่งอื่นต่อ

        
docker compose up -d --build
สั่ง build image ใหม่ก่อนเปิด container เหมาะหลังแก้ Dockerfile, package.json หรือ dependencies
ใช้เมื่อ: แก้ไฟล์สำคัญแล้วอยากให้ Docker สร้าง image ใหม่

        
docker compose down
ปิด container และลบ network ที่ Compose สร้าง แต่โดยปกติ volume ยังอยู่ ข้อมูล DB จึงยังไม่หาย
ใช้เมื่อ: เลิกใช้งานระบบชั่วคราว

      


      
🔎 ตรวจสอบสถานะและดูอาการ

      

        
docker compose ps
ดูสถานะ container ในโปรเจกต์นี้ และดู port mapping เช่น 3000->80
ใช้เมื่อ: อยากเช็กว่าระบบติดจริงไหม

        
docker compose config
ให้ Docker Compose อ่านไฟล์แล้วแสดง config ที่ถูกแปลงเรียบร้อย ถ้า YAML ผิดหรืออ้าง service ผิด มักเห็น error ก่อน up จริง
ใช้เมื่อ: อยากตรวจ config ก่อนรันระบบ

        
docker compose config --services
แสดงรายชื่อ service ที่อยู่ใต้ services: เช่น frontend, api, mongodb ชื่อเหล่านี้คือชื่อที่ใช้กับคำสั่ง Compose
ใช้เมื่อ: ไม่แน่ใจว่าต้อง stop/logs ด้วยชื่ออะไร

        
docker compose logs
แสดง log จากทุก service ใน compose ใช้ดูภาพรวมว่ามี service ไหน error
ใช้เมื่อ: ระบบเปิดไม่ติด แต่ยังไม่รู้ว่าพังตรงไหน

        
docker compose logs api
ดู log เฉพาะ service ชื่อ api ถ้า database มีปัญหา มักเห็น error จากตรงนี้
ใช้เมื่อ: API เปิดไม่ติดหรือต่อ DB ไม่ได้

        
docker compose logs -f api
-f คือ follow ดู log ต่อเนื่องเหมือน tail
ใช้เมื่อ: กำลัง debug และอยากเห็น log ใหม่ทันที

      


      
🧹 ล้าง/เริ่มใหม่

      

        
docker compose stop
หยุด container ทั้งหมดไว้เฉย ๆ ยังไม่ลบ container และ volume ยังอยู่
ใช้เมื่อ: อยากพักระบบทั้งชุดไว้ก่อน แล้วอาจกลับมา start ต่อ
